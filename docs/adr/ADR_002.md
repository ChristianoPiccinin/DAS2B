# ADR-002: Uso de Blob Storage e para armazenamento dos dados.

**Status:** Aceito | **Data:** 2026-04-07 | **Autores:** Equipe de documentação e desenvolvimento.

## Contexto
A VendaMais Distribuidora possui um sistema ERP proprietário estável para transações operacionais, porém sua camada de relatórios é limitada a exportações manuais em Excel. Esse processo gera inconsistências entre as áreas e um esforço manual significativo.

Como o ERP envia dados em formatos variados (como CSV, JSON ou exportações de tabelas), precisamos de um repositório flexível que aceite dados não estruturados antes de passarem pelo processo de limpeza e transformação. Dessa forma, a escolha para o armazenamento dos dados foi o Blob Storage, devido por armazenar grandes quantidades de dados não estruturados, além de ter um custo melhor e de se adequar mais ao Azure Functions.<br>

## Decisão
Foi decidido utilizar o Azure Blob Storage como o repositório central para as camadas Raw (Bruta) e Azure SQL Database (Processada) da plataforma.<br>
Todos os arquivos extraídos pelas Azure Functions serão organizados em containers e pastas estruturadas por data e módulo (ex: vendas/2026/04/07/pedidos.json). Isso permite que tenhamos um backup imutável de tudo o que foi extraído do ERP antes de qualquer transformação, servindo como base para auditorias e reprocessamentos futuros.

## Consequências

<span style="color:green;">**(+)**</span> **Custo-Benefício Imbatível:** O custo por GB armazenado é inferior ao de outros bancos de dados relacional, permitindo escalar o histórico da VendaMais sem preocupações financeiras imediatas.<br>
<span style="color:green;">**(+)**</span> **Escalabilidade e Flexibilidade:** Suporta volumes massivos de dados e diversos formatos de arquivo sem a necessidade de definir um esquema (*schema*) prévio.<br>
<span style="color:green;">**(+)**</span> **Desacoplamento:** Permite que a camada de ingestão termine seu trabalho rapidamente, sem depender da disponibilidade ou locks de tabelas de um banco de dados.<br>

<span style="color:red;">**(-)**</span> **Ausência de Estrutura de Consulta:** Não é possível realizar buscas complexas (SQL) diretamente nos arquivos de forma performática para dashboards.<br>
<span style="color:red;">**(-)**</span> **Complexidade de Gestão de Arquivos:** Exige uma política de nomenclatura e organização de pastas muito rígida para evitar que o storage se torne um "pântano de dados" (data swamp).<br>

## Alternativas rejeitadas
- **Enviar dados brutos diretamente para o SQL <span style="color:red;">(Rejeitada)</span>:** Sobrecarregaria o banco de dados com dados sujos e inconsistentes, além de aumentar o custo de armazenamento de forma desnecessária para informações que não precisam estar em tabelas relacionais no primeiro momento.
