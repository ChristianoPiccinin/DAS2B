# ADR-001: Estratégia de Ingestão de Dados via Azure Functions (Serverless)

**Status:** Aceito | **Data:** 2026-04-07 | **Autores:** Equipe de documentação e desenvolvimento.

## Contexto
A VendaMais Distribuidora processa aproximadamente 3.500 pedidos mensais através de um sistema ERP proprietário e estável, utilizado desde 2019. Atualmente, a diretoria toma decisões com dados que possuem até 30 dias de defasagem, e a extração de relatórios exige esforço manual intensivo.

O objetivo do projeto é automatizar a extração diária de dados transacionais (vendas, estoque, financeiro, logística) do ERP sem realizar modificações no sistema de origem ou em seu banco de dados. Precisamos de um mecanismo que conecte ao ERP de forma programada (agendada), extraia os dados do dia anterior (D-1) e os deposite em um repositório na nuvem (Azure Blob Storage), garantindo que os KPIs tenham defasagem máxima de 24 horas.<br>

## Decisão
Foi decidido utilizar Azure Functions (com a linguagem Python) operando no modelo de consumo Serverless para a camada de ingestão de dados.<br>
As funções serão configuradas com um Timer Trigger (acionador por agendamento) para rodar fora do horário comercial da VendaMais. Elas conectarão na API ou banco de dados de leitura do ERP, realizarão a extração dos blocos de dados transacionais e salvarão os arquivos brutos (camada raw) no Azure Blob Storage. O uso de Python foi escolhido devido ao seu rico ecossistema de bibliotecas para manipulação de dados.<br>

## Consequências
<span style="color:green;">**(+)**</span> **Otimização de Custos:** No modelo serverless, a VendaMais pagará apenas pelo tempo exato de execução e consumo de memória durante a extração diária, sem custos ociosos de infraestrutura.<br>
<span style="color:green;">**(+)**</span> **Escalabilidade Automática:** A infraestrutura escala nativamente. Se o volume de pedidos aumentar em datas sazonais, a função lida com o pico automaticamente sem intervenção humana.<br>
<span style="color:green;">**(+)**</span> **Facilidade de Manutenção:** O código foca apenas na lógica de extração (Python), transferindo a responsabilidade de gerenciamento de servidores e rede para a Microsoft Azure.<br>

<span style="color:red;">**(-)**</span> **Limitações de Tempo de Execução (*Timeout*):** O plano de consumo serverless possui um limite de tempo (geralmente até 10 minutos). Extrações muito grandes precisarão ser paginadas para evitar falhas.<br>
<span style="color:red;">**(-)**</span> **Cold Start:** A primeira execução após um período de inatividade pode sofrer um leve atraso na inicialização. (Risco mitigado, pois trata-se de um processamento em lote batch, e não em tempo real).<br>

## Alternativas rejeitadas
- **Azure Logic Apps <span style="color:red;">(Rejeitada)</span>:** Possui limitações severas na manipulação e transformação de grandes volumes de dados complexos em comparação ao uso de bibliotecas Python, possivelmente tendo um alto custo caso implementado nesse projeto.
